# KV 戦略

```ts
import { createKvAdapter } from '@f12o/papyr-adapter-cloudflare';

const adapter = createKvAdapter({
  kv: env.PAPYR_DOCS_KV,
  prefix: 'papyr-docs:',
});

const all = await adapter.list();
```

`prefix` を設定すると、同じ KV namespace に複数のドキュメントセットを同居させつつ、`list` のスキャン対象をその接頭辞だけに制限できます。

## KV を選ぶ場面

- とにかく最短で CRUD を動かしたい
- 1 document = 1 key のモデルで十分
- 複雑な query よりも deployment の軽さを優先したい

KV adapter は実装がいちばん素直で、PoC や小規模な公開アプリに向いています。`PapyrDocument` 全体を JSON 文字列としてそのまま保存するので、Papyr 側で追加の変換処理は必要ありません。

この docs site では公開ストレージに R2 を使っていますが、KV 戦略でも同じ `PapyrDocument` をそのまま保存できます。`PAPYR_DOCS_KV` のような binding に `prefix` を組み合わせると、単一 namespace 上でも document set を分離して運用できます。

将来的に一覧性能や query の自由度が必要になったら、同じ `BackendAdapter` interface のまま D1 に移す、という流れが取りやすいのも利点です。
